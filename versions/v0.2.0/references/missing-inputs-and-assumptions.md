# Missing Inputs and Assumptions

## Missing visual reference

Do not claim reconstruction is possible without a usable reference. Request or identify:

- the screenshot or MVP image;
- the target screen or route;
- the desired viewport;
- source assets if available.

When the environment contains no reference, produce an intake checklist rather than invented UI.

## Unreadable text

Use, in order:

1. source design or product copy supplied elsewhere;
2. existing repository copy;
3. contextually conservative transcription;
4. clearly marked placeholder of similar length.

Do not manufacture detailed product claims.

## Missing font

Search repository assets and CSS first. If unavailable, use a metrically similar legal substitute, tune line wrapping, and disclose it.

## Missing image or logo

Use, in order:

1. supplied original asset;
2. equivalent asset already in the project;
3. user-approved substitute;
4. neutral placeholder that preserves dimensions;
5. simple CSS/SVG recreation when the shape is generic and legally safe.

Never replace a distinctive brand asset invisibly.

## Unknown interaction

Implement the least surprising behavior suggested by the control’s appearance. Record it as inferred. Do not create complex backend behavior from visual evidence alone.

## Conflicting references

Rank by:

1. explicit user instruction;
2. newest approved reference;
3. reference matching the requested route and viewport;
4. existing design-system evidence;
5. conservative reconciliation.

Document the conflict and chosen authority.

## Unknown off-screen content

Build only what can be supported by references, existing code, or explicit product requirements. Do not invent an entire hidden page and call it exact.

## Unknown library

The visual output matters more than guessing the original implementation library. Reuse the current project stack and select the smallest suitable technique. Do not claim the original site used a specific library without evidence.
